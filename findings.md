## Research Findings
- Phase 19 isolated reproduction confirms the configured Codex `apply_patch` binary accepts `*** Add File: ../outside.txt`, exits 0, and creates a file outside the Gateway workspace. Canonical `apply_patch` therefore currently bypasses workspace confinement despite other file tools resolving paths safely.
- CreateDirectory/DeletePath/MovePath still perform existence checks and mutations without shared locks or parent-directory fsync; concurrent callers can observe TOCTOU-dependent generic failures.
- Phase 19 patch safety must be enforced before launching any external patch binary: parse every Add/Update/Delete/Move target, resolve through `_resolve_workspace_path`, reject escapes/symlink targets, and lock/snapshot all affected files for rollback.
- Phase 19 implementation now provides deterministic multi-path locks and durable create/delete/move helpers with parent-directory fsync and consistent not-found/conflict typing.
- `apply_patch` now rejects `..`, absolute-outside, root, symlink-component, directory, and targetless patches before process launch; locks and snapshots all Add/Update/Delete/Move targets; restores original bytes/mode and removes newly added files on non-zero exit or timeout.
- Patch subprocesses now use the bounded-output, dedicated-process-group runner with threaded stdin delivery, so custom patch binaries cannot flood Gateway memory or leave descendants after timeout.
- Adversarial regressions cover real-binary escape rejection, symlink rejection, successful patch, partial failure rollback, timeout rollback, rollback-failure disclosure, and deterministic concurrent Create/Move/Delete. Focused gates pass 13 then 14 tests.
- Phase 19 final evidence: 1131 passed, 2 skipped in 67.43s; managed real-tool acceptance passed direct runtime and native two-round orchestration; 19 deployment/config/script tests, both Compose renders, compileall, and diff-check pass.
- The confirmed apply_patch workspace escape is closed with Gateway-owned validation before external execution. Rollback covers every declared Add/Update/Delete/Move target, while undeclared side effects from a malicious administrator-configured replacement binary remain outside the trust model and are not overclaimed.
- Phase 18 live audit confirms Write/Edit/MultiEdit/RegexEdit/CopyPath/NotebookEdit perform direct in-place writes without a shared path lock, fsync, atomic replace, or existing-mode preservation. Concurrent edits can read the same source and overwrite each other.

Tests explicitly codify retrying a generic `execution_failed`; partial-side-effect tools such as apply_patch/shell/code execution can therefore be run again after a failure or timeout.

- Tool ownership is primarily risk-category based (`read_local`, `write_local`, `execute_code`, `gui`, `ai_agent` are user-side). Caller-private functions not implemented by the Gateway are also delegated rather than falsely executed.
- Workspace scoping is a first-class design: tool cache keys include both resolved workspace and runtime tenant/session scope, and orchestration runs inside a workspace scope context. This is a strong isolation design choice, provided every entry path consistently establishes the scope.
- Request concurrency uses a global `BoundedSemaphore` sized from live config, with a configurable queue timeout and deterministic release through a context manager.
- The active flow performs multiple request conversions: downstream→upstream before orchestration setup, again after planner/context manipulation, and response conversions for both extraction and downstream output. This is flexible but makes protocol idempotence and field preservation critical.
- `_execute_tool_call` permission failures fail closed. However, general non-`ToolExecutionError` exceptions return a failed `ToolResult` without recording failure/statistics, while handled transient failures are recorded after retries. This leaves an observability blind spot for common unexpected tool errors.
- Tool retry attempts have no backoff. The configured default is one retry, which is acceptable for fast transient failures but can duplicate side effects unless only explicitly retry-safe failures/tools reach that path.
- `_get_marketplace` repeats the same absolute-import issue (`from marketplace`) and silently degrades to an empty catalog, so marketplace failure can look like “no items” rather than a configuration/code error.
- Protocol conversion supports a wide Responses tool-history vocabulary (function/custom/local shell/search/computer/code-interpreter/MCP call and output variants), legacy Chat `function_call`, Anthropic `tool_use/tool_result`, and error-state preservation through a Gateway-specific content prefix.
- Gateway-only workspace/client/planner metadata is deep-copied and stripped before upstream forwarding, including nested JSON-like `metadata.user_id`. This is an important privacy and protocol-cleanliness safeguard.
- Text-tool adaptation intentionally compacts schemas and caps the injected manual. It is a pragmatic compatibility layer, but its correctness depends on robust parsing of model-generated XML/JSON/shell-like text and therefore needs stronger adversarial/fuzz testing than native protocol paths.
- There are two SQLite subsystems: `gateway_logging.py` owns request/tool/conversation operational logs, while `gateway_persistence.py` owns semantic cache/tool cache/general memories. Both use WAL, but separate connection/global initialization patterns increase lifecycle and migration complexity.
- Logging performs schema migration on startup and sanitizes imported legacy failures. The SQLite operational schema includes tenant/workspace/session indexes for conversation memory, which supports the repository's isolation goals.
- Semantic cache falls back to a custom 256-dimensional hashed character-trigram/word vector. This is cheap and deterministic, but it is lexical similarity rather than a true semantic embedding; a permissive threshold can create incorrect answer reuse. Scope keys and tool-response exclusion mitigate cross-session/tool risks.
- Static dependency analysis shows dense cycles among `gateway_tool_runtime`, `gateway_streaming`, `gateway_context`, `gateway_agent_planner`, and `gateway_builtin_tools`; several imports are deferred inside functions to survive these cycles. The facade and HTTP handler depend on 15–20 internal modules each.
- Maintainability hotspots include 493-line `do_GET`, 438-line `do_POST`, 234-line non-streaming orchestration, 203-line streaming orchestration, and multiple 300–400-line UI/config rendering functions. These are branch-heavy change-risk centers.
- In orchestrated “streaming” mode, the Gateway explicitly forces the upstream request to `stream=False`, waits for a complete response/tool round, and then synthesizes downstream SSE. Protocol shape is streamed, but token latency is not; only passthrough mode provides true upstream streaming.
- Streaming and non-streaming paths duplicate much orchestration logic (memory, planner, text fallback, ownership, tool loops, intelligence enhancement). Tests target parity, but duplication makes drift likely; for example streaming code uses broader `except Exception` handling and emits SSE errors rather than shared HTTP error types.
- HTTP Actions have meaningful SSRF defenses: scheme validation, private/non-global IP rejection, DNS resolution checks, redirect re-validation, response-size limits, and explicit per-action private-network opt-in.
- The SSRF DNS defense validates resolved addresses before urllib connects, but does not pin the validated IP to the connection; a hostile DNS service could theoretically exploit rebinding between validation and connection. This is a residual hardening concern, not evidence of an easy default bypass.
- MCP sessions are pooled per server with per-session locking, timeout-aware framed I/O, and process cleanup. Filesystem-looking downstream arguments are rejected unless explicitly allowed per server/tool, reinforcing the server/client ownership boundary.
- The Agent Planner is a deterministic, SQLite-backed state machine rather than another LLM: it classifies intent with rules, selects declared downstream tools, tracks project-analysis/fix/QA transitions, stores bounded evidence per tenant/workspace/session, and lets the weak upstream perform only final synthesis.
- This planner design gives strong tool-authority control and auditability, but `gateway_agent_planner.py` is a 4.3k-line natural-language/schema heuristic engine. New clients, languages, phrasings, or tool schemas can require fragile regex/branch changes; it should be treated as a policy compiler with contract tests, not ordinary prompt glue.
- Context fan-out runs chunk requests concurrently, tolerates individual chunk failures, synthesizes surviving partials, and optionally performs an extra review request. Partial results are appended in completion order rather than source-chunk order, although each prompt embeds its original chunk index.
- Fan-out “quality review” asks the model to review draft quality/completeness, then replaces the draft answer with the review text. Unless the model implicitly rewrites the answer, users may receive a critique instead of an improved final response. The prompt should explicitly demand a corrected final answer or the review should be used as feedback for a second rewrite.
- Fan-out with a positive `max_chunks` silently drops source text after the cap. This may be intended cost control, but the synthesis prompt/result metadata does not state that input was truncated, so the output can look complete when it is not.
- The architecture documents align with the current ownership/planner model, but the older main-entry docstring and the documented proxy “connection reuse/retry” claim do not align with the active curl implementation.
- Configuration defaults are secure for tool execution (write/shell/user-machine execution off) but permissive for service exposure: default admin password is `admin`, downstream authentication is absent unless a key is configured, and the non-production Compose publishes port 8885 on all interfaces.
- Production Compose improves this substantially by requiring upstream/admin/downstream secrets and binding the raw Gateway port to loopback. However, Nginx currently listens on port 80 only; TLS/443 directives are commented even though Compose publishes 443. Admin Basic credentials and API keys therefore travel in plaintext unless the operator manually edits Nginx exactly as the deployment guide instructs.
- Admin/downstream verifiers use unsalted single-round SHA-256. For high-entropy API keys this is generally adequate as a lookup verifier, but it is weak for human admin passwords; a slow password KDF such as Argon2id/scrypt/bcrypt/PBKDF2 should be used with migration support.
- Fernet config encryption creates a mode-0600 key in the runtime directory and encrypts nested secret-like fields. The key and encrypted config typically live in the same persistent volume, so this mainly protects accidental config disclosure, not compromise of the runtime volume/container.
- Encryption is fail-open: `encrypt_value` returns plaintext on failure, and `save_config` catches encryption errors and persists anyway. A security feature should fail closed or require explicit insecure mode; otherwise disk/key failures can silently write credentials in plaintext.
- Config saving uses `Path.write_text` without an explicit restrictive file mode. Even encrypted configuration contains hashes/topology and may be created according to a permissive umask.
- Permission-manager documentation says per-client maps are keyed by downstream key hash, but HTTP authentication passes `downstream_keys[].name` as `client_id`. An operator following the documented hash contract will miss the client-specific policy and fall through to the global default (which defaults to allow).
- Permission rules themselves are alias-aware and support global deny precedence, per-client wildcard/category rules, and deterministic default policies. The main issue is identity wiring/defaults, not the matcher.
- Primary built-in filesystem tools resolve symlinks and enforce `resolved.relative_to(workspace_root)`, which blocks ordinary `..`, absolute-path, and existing-symlink escapes. Write and shell/code tools also require explicit feature flags.
- The older `gateway_claude_compat.execute_claude_code_tool` is still publicly re-exported but implements a second, inconsistent execution stack: Write/Edit do not check `gateway.allow_write_tools`, Bash checks `tools.shell_enabled`, and Glob/Grep do not consistently resolve paths through the workspace confinement helper. It is not used by the main runtime, but public/internal callers could bypass current policy assumptions.
- This compatibility executor should be converted into format-only helpers or delegated to the canonical built-in runtime; maintaining two executable tool implementations is both a security and correctness liability.
- Canonical shell/code execution is opt-in and workspace-scoped, but captured stdout/stderr is not bounded in `_tool_shell`/`_tool_code_interpreter`. Large output can consume memory and then inflate tool rounds/logs. Long-lived exec sessions also require explicit wait/kill and lack an obvious background TTL reaper.
- Canonical file edits write in place rather than atomically. Concurrent or interrupted writes can leave partial/lost updates; this matters mostly in the explicit local-proxy mode because user-machine edits are delegated by default.
- The pytest-discoverable suite contains about 1,060 test functions/methods across 25+ files, plus numerous executable integration/smoke scripts. Coverage is broad across protocols, isolation, planner behavior, cache, permissions, persistence, concurrency, streaming, and deployment contracts.
- Test concentration is extreme: 358 tests and ~13.5k lines live in `tests/test_gateway.py`. This is a regression asset but also a refactoring bottleneck; failures and fixture state are harder to localize.
- No pytest/coverage/lint/type-check configuration or CI workflow was found in the tracked inventory. The repository relies on scripts and direct pytest invocation, so local success may not be continuously enforced in a clean environment.
- Proxy transport has only two focused tests (HTTP error detail and temp-file cleanup on timeout). There is no visible test for curl nonzero return codes/HTTP 000, active retry behavior, timeout→504 mapping, or claimed connection reuse.
- Permission tests exercise the manager in isolation, but no visible end-to-end test ties an authenticated `downstream_keys[].name` to the configured per-client permission identity contract.
- Fan-out tests check triggering, result metadata, and prompt length, but do not assert that “quality review” returns a user-ready rewritten answer rather than critique, nor disclose capped-source truncation.
- Deployment tests assert loopback binding, required production secrets, persistent runtime paths, and Nginx SSE settings, but do not require TLS to be enabled even though production Compose publishes port 443.
- Current worktree changes are small, focused hardening/persistence updates: runtime databases/anonymous workspaces/encryption keys move under configurable `GATEWAY_RUNTIME_DIR`, production raw-port binding moves to loopback, IPv6 is disabled for Compose reliability, Anthropic SSE gets an explicit Nginx route, and `cryptography` becomes a required dependency. These changes generally improve deployability/security and are not the source of the proxy/planner/compatibility issues identified above.
- The worktree also contains untracked deployment tests/design files. Analysis treats them as current code but preserves them unchanged.
- Executable verification on 2026-07-16: `python3 -m compileall -q src tests` passed; `python3 -m pytest -q` passed with **1058 passed, 2 skipped, 21 warnings in 54.15s**.

## 2026-07-17 Phase 24 Post-Streaming Remaining-Risk Audit
- The strongest remaining confirmed security gap is executable isolation. `gateway_process_ops.run_bounded_process()` launches children with the default inherited environment when no explicit `env` is supplied; Shell, Python code-interpreter, `apply_patch`, long-lived exec sessions, and Git helpers run in the Gateway container/process namespace. `_mcp_env()` explicitly starts from `dict(os.environ)`. Therefore an authorized or compromised executable can read Gateway/upstream/downstream secrets, reach the service network, mutate undeclared writable paths, and consume resources within the shared container limit. Workspace path validation, rollback, output bounds, and process-group cleanup do not provide containment against a malicious executable.
- Production Compose hardens the Gateway container itself (`read_only`, dropped capabilities, no-new-privileges, tmpfs, CPU/memory/PID limits, loopback-only raw port), but the writable workspace and `/app/data` are still shared with every in-container executable. The Admin config schema exposes `gateway.cors_enabled`, but JSON, text, error, and OPTIONS responses unconditionally send `Access-Control-Allow-Origin: *`; no origin allowlist is implemented. This should be corrected before browser-facing/external deployments, with explicit allowed origins/methods/headers and `Vary: Origin`.

A distributed lease/semaphore is still needed if `max_concurrent_requests` is intended as a deployment-wide upstream/tool capacity limit.
- Metrics are intentionally minimal: total requests, rate-limit state, and readiness. Request IDs exist in persistence, but there is no distributed span context.

## 2026-07-17 Phase 25 External-Binding/CORS Design
- Direct non-loopback listeners can be validated from the CLI host, but a container must listen on `0.0.0.0` internally even when the published host port is loopback-only. The binding contract therefore needs an explicit exposure assertion: `auto` derives from the listen host, `private` is reserved for a loopback/firewall/private-proxy publication, and `external` requires non-default Admin credentials plus at least one enabled downstream key. Ordinary Compose will assert `private` while binding only `127.0.0.1`; production Compose will assert `external` and already requires both secrets.
- External startup validation should inspect the effective decrypted configuration, not merely environment-variable presence, because a persisted config can override the generated defaults. It must reject `must_change_password`, the known `admin` verifier, and an empty/disabled downstream-key set.
- CORS should default disabled. When enabled, only exact normalized origins from `gateway.cors_allowed_origins` may receive `Access-Control-Allow-Origin`; wildcard behavior is removed. JSON, text, errors, OPTIONS, and SSE must share the same policy, and dynamic origin responses must send `Vary: Origin`.
- The existing Admin write Origin/Referer guard remains stricter than general API CORS and should continue to protect state-changing Admin routes independently.
- Phase 25 implementation now uses `GATEWAY_PUBLIC_EXPOSURE=auto|private|external`: direct non-loopback `auto` listeners fail closed, ordinary Compose is host-loopback-only plus `private`, and production Compose is `external` with required credentials.
- Explicit runtime Admin/downstream environment credentials now override stale persisted credential state in memory without writing plaintext back to disk, preserving rotation and upgrade behavior for persistent production volumes.
- CORS defaults off and exact-normalizes only HTTP(S) origins. Invalid, `null`, wildcard, disabled, and non-allowlisted preflights are rejected; allowed JSON/text/error/OPTIONS/SSE responses use the exact normalized origin and `Vary: Origin`.
- Nginx and Gateway request-body ceilings are aligned at 64 MiB. The local launcher now defaults to `127.0.0.1`, while an explicit non-loopback launch enters the external credential contract unless the operator deliberately asserts a private ingress boundary.

Compileall, diff-check, both Compose renders, two real child-process startup contracts, and checkout artifact checks passed.

## 2026-07-17 Phase 26 Worker Boundary Design
- The current bounded short-command path, long-lived exec path, Git path, and MCP path still spawn commands directly. A separate stdlib worker can safely set rlimits before `exec` without using `preexec_fn` in the multithreaded Gateway process; the worker must then remain the process-group leader so existing TERM/KILL cleanup continues to cover descendants.
- Short-lived jobs can safely receive default CPU, open-file, and output-file limits. Memory and process-count limits must remain opt-in initially because `RLIMIT_AS` and `RLIMIT_NPROC` have material platform-specific behavior, especially on macOS. Long-lived exec/MCP jobs should not inherit the short-job cumulative CPU default, but can use NOFILE/FSIZE and explicit memory/PID/CPU overrides.
- Worker setup/version/policy failure needs a reserved exit code plus a stable stderr marker. Unsupported requested limits must fail closed rather than silently launching the real command without the policy.
- Cancellation should be a first-class process-runner signal that terminates the complete process group and returns bounded stdout/stderr evidence. Timeout semantics must remain backward-compatible for existing callers.
- The worker boundary is now implemented for bounded Shell/code/apply_patch, long-lived exec, read-only Git, and MCP. It validates `gateway-sandbox-job-v1`, sets requested rlimits before `exec`, remains the process-group leader, and reserves exit 125 plus a stable marker for setup failure classification.
- Current real adversarial evidence proves CPU flood termination, `RLIMIT_FSIZE`, `RLIMIT_NOFILE`, explicit cancellation with descendant cleanup, command crash mapping, invalid-version fail-closed behavior, and MCP crash session replacement. Requested memory limits either enforce allocation failure or fail closed on platforms such as the current Darwin host where lowering `RLIMIT_AS` is rejected.
- The MCP crash regression exposed and corrected a separate stability defect: an exited worker previously leaked raw `EOFError` from the framing layer. MCP initialization and request transport failures now normalize to `ToolExecutionError`, retain sandbox setup classification, close the failed process group, and prevent session reuse.
- Network policy remains deliberately fail-closed when any value other than `inherited` is requested, because this worker batch does not yet provide a network namespace. Writable scopes are validated against workspace escape but are not yet OS-enforced; overlay/snapshot isolation remains the next material boundary.
- `apply_patch` now runs in a temporary declared-target overlay rather than mutating the original workspace directly. The Gateway rejects undeclared files/directories/symlinks, derives explicit SHA-256 create/update/delete diff entries, checks the original target bytes/modes for concurrent changes, and then performs atomic per-file replacements with transactional rollback on partial commit failure.
- Overlay regressions prove failed/timeout commands leave the original workspace untouched, undeclared overlay writes are rejected, direct external concurrent changes produce `failure_type=conflict` without being overwritten, successful moves/adds/updates still work, commit failure rolls back already committed targets, and rollback failure remains explicitly reported.
- The overlay is a strong transactional boundary for cooperative `apply_patch`, but not a complete hostile-binary sandbox: an administrator-configured malicious patch executable can still attempt an absolute path outside the overlay because the worker shares the host filesystem namespace. The same limitation remains for Shell/code/exec/MCP and must stay visible in capabilities.
- Final current batch evidence after overlay integration: full suite **1191 passed, 2 skipped in 79.05s**; compileall, diff-check, both Compose renders, and isolated local real-tool acceptance passed. The acceptance used only a local fake upstream and an isolated process-local private-network override for its localhost WebFetch fixture.

## 2026-07-17 — Phase 28 legacy SQLite compaction boundary
- An online Admin-triggered full `VACUUM` is not currently safe: normal API admission is process-local and does not cover Admin reads, the maintenance thread, or every long-lived SQLite connection, so readiness=false alone would not prove a complete write quiescence boundary.
- The safe first implementation should therefore separate surfaces: authenticated Admin may inspect compaction eligibility, while mutation is an explicit offline CLI operation requiring operator confirmation that the Gateway is stopped.
- Execution must fail closed when WAL/SHM artifacts indicate a live or unclean database, when an exclusive advisory/SQLite lock cannot be obtained, when disk headroom is insufficient, when the source signature changes, or when schema/table counts/integrity differ after compaction.
- `VACUUM INTO` plus same-directory atomic replacement can keep the original file untouched until the compacted candidate is fully validated. A temporary hard-link backup can provide rollback without duplicating the original database bytes.
- The implemented probe confirmed `PRAGMA auto_vacuum=INCREMENTAL` followed by `VACUUM INTO` produces a candidate in mode `2` while preserving table data. A 1.03 MiB fragmented fixture compacted to about 217 KiB with mode `0600`.
